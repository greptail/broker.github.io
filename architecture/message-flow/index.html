    Message Flow¶

    The broker has two primary operational modes for any queue: pass-through (direct delivery) and buffered (queue accumulation). Which path is taken depends on whether a healthy consumer exists and whether the TPS limit allows it.

    Publish Path — Producer to Broker¶

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam defaultFontName Inter
skinparam sequenceMessageAlign center

title "Publish Path — Producer → Broker"

actor "Producer" as prod
participant "QueueController\n(Spring WebFlux)" as qc
participant "BigQueueManager" as bqm
participant "QueueContainer" as qcon
participant "QueueHolder" as qh
participant "RecordLedger\n(WAL)" as rl
participant "ManagedLedgerImpl" as mli
database "Apache BookKeeper" as bk
database "MariaDB\nenqueue_record" as db

prod -> qc : POST /api/queue/publish/{queue}\n[protobuf body]
qc -> bqm : publishtoPipe(queue, message, tps)
bqm -> qcon : getQueueOrCreateIfNotExist(queue)
qcon --> bqm : QueueHolder

alt Pass-through possible\n(consumer healthy + TPS OK)
  bqm -> qh : pipe(message)
  note right of qh : skip BigQueue,\ndeliver directly
else Must buffer
  bqm -> qh : enqueue(message)
  qh -> qh : BigQueue.enqueue(bytes)
end

bqm ->> rl : writeAheadLog(LedgerLogEntry, PUBLISH)\n[async Mono, fire-and-forget]
rl -> mli : asyncAddEntry(bytes, callback)
mli -> bk : BookKeeper write

alt BookKeeper success
  bk --> mli : AddCallback.addComplete()
  mli --> rl : persist LedgerRecord in DB
else BookKeeper failure
  mli --> rl : AddCallback.addFailed()
  rl -> db : INSERT INTO enqueue_record_{queue}\n[SQL fallback]
end

qc --> prod : 200 OK (fire-and-forget)
@enduml

    Fire-and-Forget

    The HTTP response is returned to the producer before the WAL write completes. Durability is guaranteed by the async WAL — if the broker crashes before the WAL write, the message is recovered from any in-memory BigQueue state or the SQL fallback.

    Pass-Through Gateway¶

    When a consumer is available and TPS is within the configured limit, the broker acts as a transparent gateway — no buffering occurs.

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam defaultFontName Inter

title "Pass-Through Gateway Decision"

start

:Receive publish request;
:Lookup queue consumers\n(ListenerHolder);

if (Consumer registered and healthy?) then (yes)
  :Check global TPS\n(RedisGlobalRateLimiter);
  if (TPS within limit?) then (yes)
    :Check in-flight count\n(QueueHolder.inFlight());
    if (In-flight < threshold?) then (yes)
      #palegreen:Pass-through delivery\n(skip BigQueue);
      :WAL write (PUBLISH, async);
      :HTTP POST to consumer;
    else (no)
      #lightyellow:Buffer in BigQueue;
    endif
  else (no)
    #lightyellow:Buffer in BigQueue;
  endif
else (no)
  #lightyellow:Buffer in BigQueue;
endif

:Return 200 to producer;
stop
@enduml

    Consume Path — Buffered Queue Drain¶

    When messages accumulate in BigQueue (consumer down, TPS exceeded, or in-flight threshold hit), QueueHolder's scheduler continuously drains the queue and pushes to consumers.

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam defaultFontName Inter
skinparam sequenceMessageAlign center

title "Consume Path — Queue Drain → Consumer"

participant "QueueMonitorService\n(every 5 s)" as qms
participant "QueueHolder\n(Scheduler)" as qh
participant "BigQueue" as bq
participant "BigQueueManager" as bqm
participant "RecordLedger\n(WAL)" as rl
database "Apache BookKeeper" as bk
database "MariaDB\ndequeue_record" as db
participant "ListenerHolder\n(round-robin)" as lh
participant "ReactiveMessageHandler\n(WebClient)" as rmh
actor "Consumer\n(Vert.x HTTP)" as cons

qms -> qh : trigger drain if queue non-empty
qh -> qh : createScheduleIfNotExist()

loop every 1–1000 ms (adaptive)
  qh -> bq : dequeue() → bytes
  bq --> qh : BigQueueTextMessage

  qh -> bqm : handleMessage(bytes, queue)
  bqm -> rl : writeAheadLog(LedgerLogEntry, CONSUME)\n[async]
  rl -> bk : asyncAddEntry

  alt BookKeeper success
    bk --> rl : success callback
  else BookKeeper failure
    rl -> db : INSERT INTO dequeue_record_{queue}
  end

  bqm -> lh : selectConsumer() [round-robin]
  lh --> bqm : NamedConsumer(host, port)
  bqm -> rmh : handle(message, consumer)
  rmh -> cons : HTTP POST {message body}

  alt Consumer 2xx
    cons --> rmh : 200 OK
    rmh --> bqm : DeliveryAcknowledgement.SUCCESS
    note right: message confirmed delivered
  else Consumer error / timeout
    cons --> rmh : 5xx / timeout
    rmh -> rmh : retry (up to 3×)
    alt Retries exhausted
      rmh --> bqm : FAILURE
      bqm -> lh : blacklist consumer
      note right: try next consumer
    end
  end
end
@enduml

    Consumer Registration Flow¶

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "Consumer Registration"

actor "Consumer App" as cons
participant "QueueController\nPOST /subscribe/{queue}" as qc
participant "BigQueueManager" as bqm
participant "ListenerHolder" as lh
database "etcd\nConsumerRegistry" as etcd

cons -> qc : POST /api/queue/subscribe/{queue}\n{host, port, tps}
qc -> bqm : registerListener(queue, host, port, tps)
bqm -> lh : add NamedConsumer(host, port)
bqm -> etcd : register consumer in QueueConsumerRegistry
qc --> cons : 200 OK

note over lh
  ListenerHolder keeps an ordered list.
  Round-robin selects the next healthy consumer
  on each delivery attempt.
end note
@enduml

    Consumer Deregistration & Failover¶

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "Consumer Failover"

participant "ReactiveMessageHandler" as rmh
participant "ConsumerMonitorService" as cms
participant "ListenerHolder" as lh
participant "QueueFailoverService" as qfs
database "etcd" as etcd

rmh -> rmh : HTTP POST → 3× consecutive failures
rmh -> lh : blacklist(consumer)

cms -> lh : health probe (scheduled)
lh --> cms : blacklisted consumers list

cms -> qfs : initiateFailover(queue, failedConsumer)
qfs -> etcd : deregister consumer
qfs -> lh : remove from active rotation

note over lh
  Remaining consumers absorb the load.
  TPS is rebalanced across active nodes.
end note
@enduml

    End-to-End Sequence (Combined View)¶

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam defaultFontName Inter

title "End-to-End Message Flow"

actor Producer
actor Consumer

box "Broker Pod" #EEF
  participant Controller
  participant Manager
  participant QueueHolder
  participant WAL
  participant Delivery
end box

box "Storage" #FEF
  database BookKeeper
  database BigQueue
  database MariaDB
end box

Producer -> Controller : HTTP POST message
Controller -> Manager : publishtoPipe()
Manager -> WAL : WAL write (async)
WAL -> BookKeeper : asyncAddEntry

alt Consumer available + TPS OK
  Manager -> Delivery : direct push
  Delivery -> Consumer : HTTP POST
else Buffer path
  Manager -> QueueHolder : enqueue
  QueueHolder -> BigQueue : write to disk
  ...scheduler fires...
  QueueHolder -> Manager : dequeue + handleMessage
  Manager -> WAL : WAL write (dequeue)
  Manager -> Delivery : push to consumer
  Delivery -> Consumer : HTTP POST
end

Consumer --> Delivery : 200 OK
Controller --> Producer : 200 OK (fire-and-forget)
@enduml
