    System Overview¶

    Design Philosophy¶

    The broker is built around three core principles:

    1. No-shared-nothing per pod — each broker pod is fully autonomous with its own local queue storage. There is no direct pod-to-pod communication for message data.
    2. Durability before delivery — every enqueue and dequeue is logged to a distributed Write-Ahead Log (Apache BookKeeper) before the operation is acknowledged.
    3. Transparent pass-through — when a consumer is healthy and within TPS limits, the broker acts as a thin gateway: the message flows directly from producer to consumer without accumulating in the local queue.

    Component Responsibilities¶

    Ingress — QueueController¶

    REST endpoint that accepts protobuf-encoded messages from producers. Delegates immediately to BigQueueManager and returns a fire-and-forget acknowledgement. The actual durability work happens asynchronously on the WAL layer.

    Endpoint: POST /api/queue/publish/{queue}

    Core Queue Engine¶

    Class Role
    BigQueueManager Central orchestrator — publish, consume, listener registration, recovery
    QueueContainer Lazy-initialises one QueueHolder per queue name
    QueueHolder Wraps a BigQueue on disk; owns the per-queue ScheduledFuture that drains the queue and the RateLimiter for local TPS control
    ListenerHolder Tracks registered consumers for a queue; implements round-robin selection and consumer blacklisting

    WAL Layer — RecordLedger / ManagedLedgerImpl¶

    Every publish and consume event is written to a BookKeeper ledger (the WAL) before the operation is committed to the local BigQueue. This ensures that no message is lost even if the broker pod crashes mid-operation.

    ManagedLedgerImpl is modelled directly on the Apache Pulsar implementation, providing:

    • Automatic ledger rotation (by entry count, size, or time)
    • Async writes with callbacks
    • SQL fallback when BookKeeper is unavailable

    Delivery Layer — ReactiveMessageHandler¶

    A non-blocking HTTP client (Spring WebClient) that pushes messages to consumer endpoints. Each queue gets its own connection pool tuned to the queue's configured TPS:

    maxConnections  = max(tps, 2000)
pendingAcquire  = tps × 10
maxIdleTime     = 30 s
responseTimeout = 10 s

    Recovery — RecoveryService¶

    On broker restart, RecoveryService scans all ledger entries that were written but not yet fully processed (no matching dequeue record). It replays these entries to restore the correct queue state. See Recovery for the full algorithm.

    Distributed Coordination¶

    Component Mechanism Purpose
    BrokerRegistration etcd heartbeat Self-registration; other nodes discover active brokers
    BrokerCommander Redis pub/sub (broker-command topic) Broadcast delete / move / unlock commands to all pods
    RedisGlobalRateLimiter Bucket4j + Redis Enforce TPS limits across all pods for the same queue
