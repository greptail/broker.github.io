    Scaling¶

    No-Shared-Nothing Architecture¶

    Each broker pod is fully autonomous. There is no inter-pod data sharing for message storage. Pods only communicate through:

    • etcd — for node discovery and queue/consumer metadata
    • Redis — for distributed rate limiting and command broadcast
    • MariaDB — shared recovery tables (one table-set per queue)
    • BookKeeper — shared WAL (each pod writes its own ledger entries)
    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam defaultFontName Inter

title "No-Shared-Nothing Pod Topology"

package "Pod A" {
  [BigQueueManager A] as bqmA
  [BigQueue Files A\n(local disk)] as bqfA
  [QueueHolder A] as qhA
  bqmA --> qhA
  qhA --> bqfA
}

package "Pod B" {
  [BigQueueManager B] as bqmB
  [BigQueue Files B\n(local disk)] as bqfB
  [QueueHolder B] as qhB
  bqmB --> qhB
  qhB --> bqfB
}

package "Pod C" {
  [BigQueueManager C] as bqmC
  [BigQueue Files C\n(local disk)] as bqfC
  [QueueHolder C] as qhC
  bqmC --> qhC
  qhC --> bqfC
}

database "Apache BookKeeper\n(shared WAL)" as bk
database "Redis\n(rate limit + commands)" as redis
database "etcd\n(discovery + metadata)" as etcd
database "MariaDB\n(recovery tables)" as db

bqmA --> bk
bqmB --> bk
bqmC --> bk

bqmA --> redis
bqmB --> redis
bqmC --> redis

bqmA --> etcd
bqmB --> etcd
bqmC --> etcd

bqmA --> db
bqmB --> db
bqmC --> db

note bottom of bk
  Each pod writes to its
  own ledger partitions.
  No inter-pod ledger sharing.
end note
@enduml

    Horizontal Scale-Out¶

    Adding a new broker pod requires no configuration changes. The pod:

    1. Registers itself in etcd via BrokerRegistration (heartbeat every N seconds)
    2. Is discovered by other pods through NodeController (GET /api/node)
    3. Begins receiving traffic from the load balancer immediately
    4. The global rate limiter in Redis automatically distributes TPS across all active pods
    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "New Pod Joins Cluster"

participant "New Pod" as np
participant "etcd" as etcd
participant "Load Balancer" as lb
participant "Redis\nRate Limiter" as redis
participant "Other Pods" as pods

np -> etcd : register heartbeat\n(BrokerRegistration)
etcd --> pods : pod-list updated
lb -> np : begin routing traffic
np -> redis : participate in global TPS bucket
note over redis
  Per-queue TPS is split across active pods:
  effectiveTPS = configuredTPS / activePodCount
end note
@enduml

    Queue Instance Tracking¶

    A queue can be active on one pod at a time for a given partition. QueueInstanceTrackingRegistry in etcd ensures only one pod owns a queue instance:

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "Queue Instance Ownership"

participant "Pod A" as podA
participant "Pod B" as podB
database "etcd\nQueueInstanceTrackingRegistry" as etcd

podA -> etcd : acquire lock for queue:orders:partition-0
etcd --> podA : GRANTED

podB -> etcd : acquire lock for queue:orders:partition-0
etcd --> podB : DENIED (already owned)

note over podA, podB
  Pod B routes publishes to Pod A,
  or waits for lock expiry/release.
  Prevents duplicate delivery from two pods.
end note
@enduml

    TPS Distribution Across Pods¶

    The global rate limiter uses Redis + Bucket4j to enforce a per-queue TPS limit globally, even when multiple pods serve the same queue:

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "Global TPS Rate Limiting"

actor "Producer" as prod
participant "Pod A\nRateLimiter" as podA
participant "Pod B\nRateLimiter" as podB
participant "Pod C\nRateLimiter" as podC
database "Redis\nBucket (queue:orders, 1000 TPS)" as redis

prod --> podA : request
prod --> podB : request
prod --> podC : request

podA -> redis : tryConsume(1)
podB -> redis : tryConsume(1)
podC -> redis : tryConsume(1)

redis --> podA : OK (bucket has tokens)
redis --> podB : OK
redis --> podC : REJECTED (bucket empty)

podC --> prod : 429 Too Many Requests

note over redis
  All pods share one bucket per queue.
  Total across all pods never exceeds
  the configured TPS limit.
end note
@enduml

    Kubernetes Deployment¶

    The broker is deployed as a Deployment (or StatefulSet for stable pod identity) in Kubernetes. Local BigQueue files are backed by a PersistentVolumeClaim per pod.

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "Kubernetes Topology"

package "Kubernetes Cluster" {

  package "broker Deployment" {
    rectangle "broker-pod-0\n(PVC: queue-data-0)" as p0
    rectangle "broker-pod-1\n(PVC: queue-data-1)" as p1
    rectangle "broker-pod-2\n(PVC: queue-data-2)" as p2
  }

  rectangle "Service\n(ClusterIP / LoadBalancer)" as svc
  rectangle "HPA\n(CPU / custom metrics)" as hpa

  svc --> p0
  svc --> p1
  svc --> p2
  hpa ..> p0 : scale out/in
}

cloud "BookKeeper Ensemble" as bk
cloud "Redis Sentinel/Cluster" as redis
cloud "etcd Cluster" as etcd
cloud "MariaDB (Galera)" as db

p0 --> bk
p1 --> bk
p2 --> bk
p0 --> redis
p1 --> redis
p2 --> redis
p0 --> etcd
p1 --> etcd
p2 --> etcd
p0 --> db
p1 --> db
p2 --> db
@enduml

    Helm Values¶

    The broker ships with a Helm chart under helmchart/. Key scaling values:

    replicaCount: 3

resources:
  requests:
    cpu: "500m"
    memory: "1Gi"
  limits:
    cpu: "2"
    memory: "4Gi"

persistence:
  enabled: true
  storageClass: "fast-ssd"
  size: "50Gi"

autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70
