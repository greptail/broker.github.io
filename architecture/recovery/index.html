    Recovery¶

    When a broker pod restarts, it must reconstruct the correct queue state from the Write-Ahead Log. The recovery process ensures that no messages are lost and no messages are delivered twice.

    Recovery Overview¶

    The broker uses a two-tier WAL approach for recovery:

    1. BookKeeper ledgers — primary source of truth (expensive, high-performance)
    2. MariaDB tables — secondary source (cheap, persistent, easier to query)

    During normal operation, every enqueue and dequeue is written to both a BookKeeper ledger entry and (on success) to SQL recovery tables. On restart, the broker reconciles these two sources to determine what messages are still in the queue.

    Storage Migration: Expensive → Cheap¶

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam defaultFontName Inter

title "Ledger-to-SQL Storage Migration"

start

:Broker running normally;

repeat
  :RecoveryService scheduled scan;
  :Iterate each open ledger for queue;
  :Read ledger entries;

  fork
    :Extract PUBLISH entries;
    :Write to enqueue_record_{queue};
  fork again
    :Extract CONSUME entries;
    :Write to dequeue_record_{queue};
  end fork

  :Mark ledger as processed\n(LedgerTrackingRegistry);
  :Ledger can now be garbage collected;
repeat while (more ledgers?)

stop
@enduml

    BookKeeper ledgers are expensive storage — they occupy bookie disk space and require an ensemble. Once processed into MariaDB tables, the ledger data moves to cheap storage (relational rows) and the ledger itself can be deleted.

    Restart Recovery Algorithm¶

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam defaultFontName Inter

title "Broker Restart Recovery"

start

:Pod starts;
:RecoveryService.init();

:Query LedgerTrackingRegistry\nfor unprocessed ledgers;

if (Unprocessed ledgers exist?) then (yes)
  repeat
    :Open ledger from BookKeeper;
    :Read all entries;

    fork
      :PUBLISH entries →\nINSERT INTO enqueue_record_{queue};
    fork again
      :CONSUME entries →\nINSERT INTO dequeue_record_{queue};
    end fork

    :Mark ledger as processed;
  repeat while (more unprocessed ledgers?)
else (no)
  :All ledgers already migrated to SQL;
endif

:Reconcile SQL tables;
note right
  Messages remaining =
  enqueue_record rows with no
  matching dequeue_record row
end note

:Remaining rows = actual queue state;
:Re-enqueue into local BigQueue;

:Queue ready for production;
stop
@enduml

    SQL Table Reconciliation¶

    The recovery logic is essentially a set difference:

    Queue State = enqueue_record_{queue}  EXCEPT  dequeue_record_{queue}

    Messages in enqueue_record but not in dequeue_record were produced but never successfully consumed. They must be re-enqueued.

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "Recovery Reconciliation"

rectangle "enqueue_record_{queue}" as er {
  card "msg-001 (enqueued)" as e1
  card "msg-002 (enqueued)" as e2
  card "msg-003 (enqueued)" as e3
  card "msg-004 (enqueued)" as e4
}

rectangle "dequeue_record_{queue}" as dr {
  card "msg-001 (consumed)" as d1
  card "msg-003 (consumed)" as d3
}

rectangle "Recovered Messages\n(re-enqueue to BigQueue)" as rm {
  card "msg-002" as r2 #palegreen
  card "msg-004" as r4 #palegreen
}

e1 ..> d1 : matched — skip
e2 --> r2 : no match — recover
e3 ..> d3 : matched — skip
e4 --> r4 : no match — recover
@enduml

    Consumer-Available Recovery Path¶

    When a consumer is registered for a queue at the time of recovery (rather than loading into BigQueue), messages can be delivered directly from the SQL table using a DELETE … RETURNING pattern:

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA
skinparam sequenceMessageAlign center

title "SQL-Driven Consumer Delivery (Recovery)"

participant "RecoveryService" as rs
participant "QueueMonitorService" as qms
database "MariaDB\nenqueue_record_{queue}" as db
participant "ReactiveMessageHandler" as rmh
actor "Consumer" as cons

rs -> db : find unmatched enqueue_record rows
rs -> qms : notify: queue has residual messages
qms -> rs : consumer available?

alt Consumer registered
  loop for each residual message
    rs -> db : DELETE FROM enqueue_record_{queue}\nWHERE id = ?\nRETURNING message_payload
    db --> rs : message_payload (atomic — no double delivery)
    rs -> rmh : deliver message to consumer
    rmh -> cons : HTTP POST
    cons --> rmh : 200 OK
    note right: row deleted atomically;\nno re-delivery on crash
  end
else No consumer
  rs -> rs : enqueue to local BigQueue\nfor later delivery
end
@enduml

    The DELETE … RETURNING pattern is atomically safe: if the broker crashes between the delete and the HTTP POST, the row is gone and the message is lost. To guard against this, the message is written to BigQueue before the delete in non-consumer-available paths.

    Queue Deletion Cleanup¶

    @startuml
!theme plain
skinparam backgroundColor #FAFAFA

title "Queue Deletion — Distributed Cleanup"

actor "Admin" as admin
participant "QueueController\nDELETE /api/queue/{name}" as qc
participant "BigQueueManager" as bqm
participant "BrokerCommander\n(Redis pub/sub)" as bc
participant "All Broker Pods" as pods
database "etcd" as etcd
database "MariaDB" as db
database "BigQueue Files" as bqf

admin -> qc : DELETE request
qc -> bqm : requestQueueDeletion(queueName)
bqm -> etcd : mark queue for deletion
bqm -> bc : broadcast DeleteCommand\n(broker-command topic)

bc ->> pods : DeleteCommand received

pods -> bqm : purgeQueue(queueName)
bqm -> bqf : delete local BigQueue files
bqm -> db : DROP TABLE enqueue_record_{queue}
bqm -> db : DROP TABLE dequeue_record_{queue}
bqm -> etcd : remove queue metadata
@enduml

    Recovery Service Configuration¶

    Parameter Default Description
    bigbro.broker.recovery.timeout 2 Dequeue timeout in seconds when scanning for residuals
    Scan interval 1 s bufferRecovery() scheduled task frequency
    Ledger stats cache 5 s collectPendingLedgerStats() refresh interval

    Pending Message Count¶

    RecoveryService.pendingMessageToBeRecovered() counts unprocessed ledger entries across all queues. This is used by:

    • QueueMonitorService to decide whether to trigger recovery
    • AggrigatedStatisticService to expose recovery backlog metrics
    • Liveness/readiness probes to delay traffic until recovery completes
